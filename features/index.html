Feature guide

Everything MIYO adds around reading, downloads, and local control.

This page turns the README-level overview into a user-facing guide for what the app is meant to do and where each feature fits.

Reading Downloads Plugins Native

Offline-first reader behavior

MIYO is built for users who keep large local libraries and expect downloaded chapters to open without wasting network data.

  • Downloaded pages are preferred before online retrieval.
  • Local storage and history paths are treated as core workflows, not side features.
  • Reader state is designed around progress, page indexing, and resume behavior.

Library surfaces

The app keeps history, favorites, local storage, explore, and feed surfaces separate so users can find content by intent instead of digging through one overloaded screen.

  • History for recently read titles.
  • Favorites and categories for owned collections.
  • Local storage for imported and downloaded chapters.

Download reliability

Downloads are designed around real mobile conditions: slow providers, background work, notification controls, cancellation, and partial files.

  • Background worker flow with user-visible progress.
  • Stuck-state diagnosis hooks for long-running downloads.
  • Safer local chapter validation before declaring work complete.

Refinement and storage

Optional image refinement is treated as a local post-processing workflow. The app avoids deleting original files until refined output is validated as readable.

  • Refinement models are distributed separately when they should not bloat the main app.
  • Image and archive work is guarded against corrupted output.

Source plugin boundary

MIYO keeps source-specific parsing outside the core reader where possible. Compatible plugins are currently Kotatsu-style JAR parser plugins.

  • GitHub release imports for compatible JAR plugins.
  • Keiyoushi/Mihon APK repositories are detected and blocked until a real compatibility runtime exists.

Captcha and provider realities

Some providers require browser or Cloudflare verification. MIYO exposes these flows in the app while keeping provider behavior separate from the website.

This website does not proxy, mirror, or unlock any provider content.

C++ integrations

Native helpers are used where they can reduce app latency or make image/archive work more predictable.

  • Image probing and format inspection.
  • Archive-oriented native paths where Android/Kotlin overhead is avoidable.
  • Future refinement acceleration without moving the whole app into native code.

Design rule

Native code should support the reader. It should not become a second app hidden inside the app. Keep the boundary clear, measurable, and recoverable.
